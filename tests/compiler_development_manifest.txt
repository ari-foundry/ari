# Compiler development gate manifest
#
# This manifest is checked by tests/check_compiler_development_docs.py. It keeps
# Ari's compiler-development work split by layer instead of collapsing into one
# vague "bootstrap" task. Entries name the gate and the test style that should
# eventually protect it.

frontend-grammar: ok and error fixtures for lexer/parser syntax surfaces
development-dashboard: one-page compiler-development status, next actions, small checks, and readiness seed
contributor-guide: edit map, small-check loop, test categories, natural Ari design rules
concepts-glossary: compiler layer terms, artifact vocabulary, review language, and ownership boundaries
layer-map: source files, layer ownership, first artifacts, docs, and focused checks
triage-guide: symptoms, first owning layer, artifact choice, test bucket, and smallest check
source-identity-authoring: source kinds, stable source ids, byte spans, line-column lookup, and source-map policy
module-project-authoring: file modules, roots, search paths, metadata, caches, and module graph policy
artifact-authoring: artifact order, golden files, normalization, focused checks, and executable-last policy
diagnostic-authoring: diagnostic code families, messages, labels, notes, source spans, and golden tests
test-authoring: fixture buckets, behavior-based file names, expected results, and artifact update policy
implementation-playbook: small compiler implementation slices, first files, first checks, and review criteria
next-slices: near-term compiler-development queue and first artifact choices
change-checklist: review checklist for docs, tests, diagnostics, sema, IR, and non-goals
readiness-inventory: strengths, blocking gaps, development backlog, bootstrap start gate
readiness-scorecard: weighted readiness score and current percent estimate as ordinary Ari data
source-identity: SourceId, SourceFile, Span, byte range, line and column lookup
diagnostics: diagnostic codes, labels, notes, stable golden rendering
pass-contracts: lexer, parser, resolver, HIR, sema, typed IR, backend boundaries
source-diagnostics-layer: compiler/source and compiler/report packages outside runtime std
module-projects: file-backed modules, roots, visibility, cache invalidation
project-module-model: file-backed modules, project roots, search paths, metadata, module cache policy
generic-models: nested structs, enums, Vec, maps, sets, Result payloads
trait-selection: Drop, Debug, formatting, Eq, Ord, Hash dispatch
error-flow: Result-based expected failures without panic-heavy paths
allocation-model: zones, scratch arenas, ownership, drop, borrow diagnostics
ir-contract: resolved sema facts lowered into typed IR
backend-artifacts: LLVM IR, object, executable, shared-library artifact checks
tool-build-flow: focused Make targets for one Ari tool and its fixtures
artifact-testing: capability inventory, token, diagnostic, syntax, HIR, typed IR, LLVM, object, executable comparison policy
test-classification: ok, errors, artifact, tooling, backend, and executable fixture categories
doc-crosswalk: language feature docs, examples, test families, and small checks
policy-errors: class/interface bootstrap-only syntax stays rejected in compiler-development fixtures
stage-comparison: capability inventory, token, syntax, HIR, typed IR, LLVM text comparison order
